# 知识库 - 系统需求 (SR) 编写规范

* **版本状态**: Active
* **适用范围**: 所有跨系统交付与前后端接口对接规范
* **数据来源**: 软件系统工程最佳实践
* **更新时间**: 2026-07-11

## 核心定义
**系统需求（System Requirement，SR）** 是指将产品业务需求（PRD）翻译为开发可直接落地、测试可直接测量的系统内部及系统间协同的底层技术要求描述。

## SR 编写的核心要素

1. **确定系统拓扑与边界**：
   - 清楚画出数据和信令流经的系统。例如：客户端（Client） → 网关（Gateway） → 鉴权服务（Auth Service） → 数据库（DB）。
2. **规范时序交互流程 (Sequence)**：
   - 使用标准 Mermaid 或时序图，画出并发起各调用。
   - 标明是同步调用（Sync）还是异步调用（Async），写清超时判定时长。
3. **严格的接口约束 (Schema)**：
   - 数据字段类型必须标明（如 Integer, String, Boolean）。
   - 数据格式规范限制（如 电话号码必须符合正则 `^1[3-9]\d{9}$`）。
   - 数据安全级别（如 密码字段必须以哈希值存储，传输中必须加密，禁止明文流转）。
4. **异常场景闭环校验**：
   - 针对无网、弱网、并发拥堵、越权请求、Token 失效等全部技术异常分支，必须声明明确的错误码（ErrorCode）与降级（Fallback）响应机制。
