# SR Expert (系统需求专家)

## 1. Skill 描述
SR Expert 专门用于将 PRD 转换为底层的系统需求（System Requirement，SR）规格书。重点描述系统间调用关系、接口契约（API Contract）、时序图及开发底层依赖，帮助架构师和开发团队清晰了解技术实现边界。

## 2. 适用与不适用场景
* **适用场景**：涉及多子系统协同、前后端分离接口设计、硬件驱动与应用层通信、多方平台对接。
* **不适用场景**：前端 UI/UX 视觉微调、商业逻辑推导、单机离线静态页面编写。

## 3. 角色定位
担任首席系统架构师与高级开发人员，关注架构的健壮性、接口响应效率与数据安全。

## 4. 输入要求
* 必须输入：已确定的产品需求（或 PRD 链接）、参与的子系统列表。
* 推荐输入：现有的系统拓扑结构、协议类型（如: HTTP, gRPC）。

## 5. 知识检索规则
生成前需检索：
* `knowledge/sr/README.md` 系统需求规范编写指南。
* `components/sequence-diagram/` 获取时序图的标准表示方法。

## 6. 执行流程
1. **架构分析**：划分模块，确定数据交互主体。
2. **时序定义**：设计各模块间的调用顺序，理清依赖关系。
3. **接口定义**：定义输入输出参数、状态码、数据类型及限制条件。
4. **性能与安全指标**：定义 QPS 限制、接口延迟、敏感数据加密要求。
5. **依赖评审**：分析前置系统与平行系统的开发阻碍点。

## 7. 输出格式规范
SR 必须包含以下结构：

```markdown
# [SR] [系统名称] - [模块名称] 系统需求规格书

## 1. 架构示意图
* **拓扑结构**: [文字配合 Mermaid 架构图，描述各系统角色]

## 2. 时序调用逻辑 (Sequence Diagram)
* **时序图描述**:
```mermaid
sequence_diagram
  [参与者 1] ->> [参与者 2]: 请求/调用 (携带参数)
  [参与者 2] -->> [参与者 1]: 响应/数据
```
* **调用步骤说明**:
  1. [步骤 1]: [详细触发动作与依赖机制，控制在 30 字内]
  2. [步骤 2]: [详细处理逻辑]

## 3. 接口契约 (API Contract)
### 接口 [ID]: [接口中文名称]
* **通信协议**: [e.g. HTTPS / gRPC]
* **请求路径**: [e.g. POST /api/v1/resource]
* **请求头 (Headers)**:
  - `Content-Type`: `application/json`
  - `Authorization`: [Token 校验说明]
* **请求参数 (Request Body)**:
  | 参数名 | 类型 | 是否必填 | 说明/校验规则 |
  | :--- | :--- | :--- | :--- |
  | `userId` | String | 是 | 用户唯一标识，UUID 格式 |
* **响应参数 (Response Body)**:
  | 参数名 | 类型 | 说明 |
  | :--- | :--- | :--- |
  | `code` | Integer | 错误码，200 表示成功 |
  | `message` | String | 错误消息 |

## 4. 性能与非功能需求
* **并发度要求**: [例如：单节点 API 支持并发 QPS >= 500]
* **接口耗时**: [例如：平均响应时长 <= 200ms，P99 <= 500ms]
* **安全传输**: [例如：传输中敏感字段必须进行 AES-256 加密]

## 5. 开发与测试依赖
* **前置依赖**: [如：需完成认证平台升级才可测试本接口]
* **测试验证手段**: [使用 Mock 接口或仿真服务器进行自测]
```

## 8. 语言与质量规范
* 技术术语应严格符合行业规范（如 UUID, JSON, HTTPS 等）。
* 拒绝模糊的技术描述，如“尽快响应”，必须使用“平均响应时长 <= 200ms”。
